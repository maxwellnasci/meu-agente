## Debug realizado — 4 causas raiz encontradas e resolvidas

1. Docker CLI ausente na imagem reconstruída
   Causa: rebuild sem --build-arg OPENCLAW_INSTALL_DOCKER_CLI=1
   Solução: sempre incluir esse argumento em rebuilds
   (já documentado em docs/MANUTENCAO.md)

2. Fila de entrega "durable" travando silenciosamente
   Causa: delivery.durable desviava para mecanismo complexo
   de reconciliação multi-conta
   Solução: usar entrega direta via deliver()

3. Token de acesso temporário expirado
   Causa: token via popup OAuth dura ~1-2h (mais curto que
   o token padrão de 24h do painel API Setup)
   Solução: gerado token PERMANENTE via System User
   (Business Settings → System Users → gerar token com
   permissões whatsapp_business_messaging +
   whatsapp_business_management, sem expiração)

4. Normalização de números brasileiros
   Causa: Meta envia webhook SEM o 9 no número, mas exige
   o 9 para enviar mensagem de resposta
   Solução: normalização específica em phone.ts

## Configuração final do allowFrom
Mantidos os dois formatos do número (com e sem o 9). A
autorização de mensagens recebidas usa o número exatamente
como a Meta entrega no webhook (sem o 9); o envio de
resposta normaliza automaticamente para o formato com o 9
exigido pela API. Remover a variante sem o 9 bloquearia
mensagens futuras do próprio número autorizado.

## Lições aprendidas

### Lição 1: Token expirado pode disfarçar de bug de código
Um travamento sem erro claro levou a horas de debug em
sandbox/kernel antes de descobrir que era token vencido.
Lição: sempre verificar validade de credenciais ANTES de
investigar código quando algo que funcionava para de
funcionar do nada.

### Lição 2: Nem todo "token temporário" da Meta dura 24h
Token gerado via popup OAuth (fluxo de "Gerar token de
acesso" com seleção de contas) dura ~1-2h, mais curto que
o token do painel API Setup (~24h). Para produção, sempre
usar token permanente via System User.

### Lição 3: Canal real > TaskFlow para conversas com memória
A escolha de implementar como canal real (igual Telegram/
Discord/SMS) ao invés de rota HTTP + TaskFlow foi decisiva
para o Amigão manter contexto entre mensagens - essencial
para atendimento de verdade.

### Lição 4: Debugging sistemático encontra causa raiz real
4 problemas em sequência (Docker CLI, fila de entrega,
token, formatação de número) foram isolados um por um,
com testes diretos (bypass do framework) usados para
separar "bug de código" de "problema de credencial/config".

## Correção (mesmo dia, à tarde): o "token System User" de mais cedo não era de verdade

O token registrado acima (linha "Token: PERMANENTE via System User") e a
Lição 2 estavam **incorretos**. O token gerado no início do dia parecia
permanente mas expirou em ~11h (validado hoje à tarde contra
`https://graph.facebook.com/debug_token`, retornando
`OAuthException code 190 / subcode 463 - Session has expired`).

Causa raiz: o token foi gerado pelo painel **WhatsApp → API Setup**
(dentro do produto WhatsApp no Meta for Developers), não pelo caminho
correto de System User. Esse painel emite tokens de curta duração
mesmo quando a interface não deixa isso óbvio.

Caminho correto (confirmado funcionando):
**Business Settings → Users → System Users → Generate Token**,
selecionando o app correto e marcando as permissões
`whatsapp_business_messaging` + `whatsapp_business_management`,
com expiração "Nunca".

Token novo gerado por esse caminho e validado via `debug_token`:
```json
{
  "type": "SYSTEM_USER",
  "is_valid": true,
  "expires_at": 0,
  "scopes": ["whatsapp_business_management", "whatsapp_business_messaging",
             "whatsapp_business_manage_events", "public_profile"]
}
```

**Lição 2 revisada:** nem todo token que "parece" System User é
System User de verdade. O único jeito confiável de confirmar é rodar
`debug_token` e checar `type: SYSTEM_USER` + `expires_at: 0` — não
basta a impressão de que o token veio de um fluxo "permanente". O
caminho de geração importa: sempre Business Settings → Users →
System Users → Generate Token, nunca pelo painel de API Setup do
produto WhatsApp.
