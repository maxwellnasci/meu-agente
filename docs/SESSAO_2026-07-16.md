# Sessão 2026-07-16 — Auditoria do fix do Bug 4 (proposto pelo Antigravity)

## Contexto

O Antigravity (Sonnet 4.6 think) investigou o Bug 4 numa sessão separada,
concluiu que a causa era o hook `reply_dispatch` (registrado pelo plugin
`acpx`) sem timeout, disparando `acpManager.runTurn()` (rede/subprocesso)
sem proteção. Aplicou um fix em `openclaw/src/plugins/hooks.ts` (2 commits:
`12c785e9bf` código + doc `METODO_DISJUNTOR_SUBAGENTES.md`), mas **nunca
rodou o teste que ele mesmo escreveu** — esbarrou no mesmo
`ERR_NO_TYPESCRIPT` do Node do host já documentado na Retomada 1
(`docs/SESSAO_2026-07-15.md`). A conclusão veio de subagentes lendo código,
não de reprodução real. Pedido explícito do usuário: auditar com rigor
antes de aceitar como resolvido — mesma disciplina já usada com a hipótese
do DeepSeek (Retomada 3).

## Achado 0 (urgente, fora do escopo original) — produção estava com a
combinação de risco ligada, sem estar no plano documentado

Antes de auditar o fix, checagem do estado real de produção
(`/home/max/.openclaw/openclaw.json`) mostrou `plugins.entries.
github-repo-report.enabled: true` e `tools.sandbox.tools.alsoAllow:
["github_repo_report"]` — a mesma combinação que causou o Bug 4 original.
O arquivo tinha `mtime` de **23:00 do dia 15/07**, exatamente o timestamp
do commit do fix — indicando que alguém (Antigravity ou o usuário testando
a correção) reconectou a tool em produção durante a investigação, sem isso
aparecer em nenhum registro de sessão.

**Mais grave:** o container rodando (`openclaw-openclaw-gateway-1`) usava
imagem Docker criada às **22:22** do dia 15/07 — **antes** do commit do fix
(23:00). Ou seja: produção estava rodando, ao vivo, com o WhatsApp real
conectado, a tool de risco habilitada e **sem** o fix. Confirmado por
leitura direta do bundle minificado dentro do container
(`grep resolve_exec_env` presente, sem entrada equivalente para
`reply_dispatch`/`reply_payload_sending`). `healthz` respondia OK e CPU
estava ociosa no momento da checagem (sem travamento em andamento), mas o
risco de uma nova mensagem tipo "analisa o mox" travar de novo estava live.

**Ação tomada (autorizada pelo usuário):** editado
`/home/max/.openclaw/openclaw.json` removendo `plugins.entries.
github-repo-report` e `tools.sandbox.tools.alsoAllow`, restaurando o
estado seguro documentado (só `plugins.entries.deepseek`). Aplicado via
`docker compose restart openclaw-gateway` (o hot-reload do gateway já
tinha pego a mudança antes mesmo do restart, pelos logs). Confirmado:
`healthz` → `{"ok":true,"status":"live"}`, log do gateway caiu de "9
plugins" (com `github-repo-report`) pra "8 plugins" (sem ela).

**Lição para próximas investigações (pedida explicitamente pelo usuário):**
mudanças temporárias de config feitas *durante* uma investigação (pra
testar uma hipótese, reproduzir um bug etc.) precisam ser revertidas **na
hora**, não só "no final da sessão" — porque a sessão pode terminar (ou ser
interrompida) no meio, deixando produção exposta sem que ninguém tenha
decidido isso conscientemente. Ver `[[feedback_revert_temp_config_immediately]]`.

## Achado 2 — testes rodados de verdade pela primeira vez (via Node oficial em container, não host)

Ambiente já documentado (Retomada 1): Node do Kali via `apt` não tem
suporte nativo a TS (`process.features.typescript === false`), quebra
qualquer script `.ts`. Caminho que funciona: rodar dentro de
`node:24-bookworm` oficial (mesma imagem que o `Dockerfile` usa pra build),
montando o repo via bind mount — sem precisar buildar a imagem inteira da
aplicação, só reaproveitando o `node_modules` já instalado no host (pnpm,
sem rebuild de dependências).

```
docker run --rm -v "$(pwd)":/app -w /app -e OPENCLAW_VITEST_MAX_WORKERS=1 \
  node:24-bookworm bash -c "corepack enable; node scripts/run-vitest.mjs <path>"
```

- Teste novo do fix (`hooks.reply-dispatch-timeout.test.ts`): **passou**
  (1/1), primeira vez que ele roda de verdade.
- Suíte completa de testes de `hooks.ts` (13 arquivos, todos os
  `hooks.*.test.ts`): **77/77 passaram**, sem regressão em nenhum outro
  hook (`before_agent_start`, `before_agent_finalize`, `before_agent_reply`,
  `compaction`, `security`, `phase-hooks` etc.).

## Achado 4 (decisivo) — `acpx` nunca esteve instalado neste deployment

Este é o achado que muda a conclusão inteira. Checado o log real do
gateway (`docker logs openclaw-openclaw-gateway-1`):

```
- plugins.entries.acpx: plugin not installed: acpx — install the
  official external plugin with: openclaw plugins install @openclaw/acpx
```

Essa linha aparece quando alguém tentou habilitar `acpx` via
`plugins.entries.acpx` durante a madrugada de hoje (01:44-02:05,
16/07) — o gateway **rejeitou**, porque o plugin não está instalado.

Mais importante: a lista de plugins carregados **no exato momento do
incidente original** (2026-07-15, 22:22:53 até 22:34:15, janela que cobre
o "analisa o mox" travado às 22:23:28-22:23:34, ver `docs/
SESSAO_2026-07-15.md` Retomada 2) foi:

```
9 plugins: browser, canvas, device-pair, file-transfer,
github-repo-report, memory-core, phone-control, talk-voice, whatsapp-cloud
```

**Sem `acpx` em lugar nenhum.** O único handler de produção que registra
`reply_dispatch` é `tryDispatchAcpReplyHook`
(`extensions/acpx/index.ts:20`, via `api.on("reply_dispatch", ...)`). Se
`acpx` nunca carregou, esse handler nunca foi registrado, e
`getHooksForName(registry, "reply_dispatch")` retorna lista vazia —
`runClaimingHook` (`hooks.ts:699-701`) devolve `undefined` **imediatamente**,
sem possibilidade de travar por esse caminho.

**Conclusão: o mecanismo que o fix protege não pode ter causado o Bug 4
real, porque esse mecanismo nunca esteve ativo neste ambiente.**

## Achado 5 — o suspeito que sobra (já identificado, nunca corrigido)

A Retomada 7 (`docs/SESSAO_2026-07-15.md`) já tinha listado **dois**
suspeitos de "perigo alto", não um: `reply_dispatch` + `acpx` (endereçado
pelo fix, mas inerte aqui) e **`before_agent_reply` + `memory-core`**
(não endereçado). `memory-core` **está** ativo em produção (presente em
todo boot log, com ou sem `github-repo-report`). Confirmado o registro
real: `extensions/memory-core/src/dreaming.ts:929`,
`api.on("before_agent_reply", async (event, ctx) => {...})`, que faz
`reconcileManagedDreamingCron` + `runShortTermDreamingPromotionIfTriggered`
(I/O de disco, imports dinâmicos paralelos, spawn de subagente) — sem
timeout próprio, e `before_agent_reply` **não está** no mapa de timeout
default (nem antes nem depois do fix do Antigravity).

Atenuante real (leitura do código, linhas 931-933): esse handler só faz o
trabalho pesado se `ctx.trigger === "heartbeat"` ou `"cron"` — pra uma
mensagem direta do WhatsApp ("analisa o mox"), o trigger normal não é
nenhum dos dois, então o handler deveria retornar `undefined` cedo na
maioria dos casos. Isso enfraquece (mas não elimina) a hipótese pra esse
incidente específico — `cron.enabled: false` no `openclaw.json` atual, mas
não foi confirmado se o heartbeat dispara independente disso. **Não
investigado a fundo nesta sessão** — decisão consciente de fechar a
auditoria do fix do Antigravity primeiro, já que essa é uma pista nova,
não uma correção pendente de decisão já tomada.

## Conclusão

**Bug 4 não está resolvido.** O commit `12c785e9bf` é uma correção
tecnicamente correta e bem-testada (confirmado agora, não só assumido) —
mas protege um caminho (`acpx` → `reply_dispatch`) que nunca esteve ativo
neste ambiente, então não pode ter sido a causa do travamento real
observado em 2026-07-15. É hardening de framework válido (útil se
`acpx` for instalado no futuro), não o fix do Bug 4.

Mesmo padrão de falha da hipótese do DeepSeek (Retomada 3): conclusão
construída por leitura de código plausível, sem confirmar contra o estado
real do sistema (nesse caso, "esse plugin está de fato carregado?").
Lição geral: **antes de atribuir um hang a um hook/plugin, confirmar contra
o log real do gateway (`N plugins: ...`) quais plugins estavam carregados
no momento do incidente — não basta o plugin existir em `extensions/`.**

## Retomada — `before_agent_reply` + `memory-core` também refutado

Investigação pedida explicitamente antes de aceitar esse suspeito (mesma
disciplina do resto da auditoria): confirmar se ele de fato dispara e faz
trabalho pesado neste deployment, não só se o código parece perigoso.

**Pergunta 1 — heartbeat dispara independente de `cron.enabled`?** Sim, mas
irrelevante pro caso: `startHeartbeatRunner` (`src/gateway/
server-runtime-services.ts:229`) é conectado incondicionalmente no boot,
módulo separado do `CronService`/`cron.enabled` (zero referências cruzadas
em `heartbeat-runner.ts`). Sem config de heartbeat no `openclaw.json`,
cai no default `DEFAULT_HEARTBEAT_EVERY = "30m"` (`src/auto-reply/
heartbeat.ts:24`), habilitado por padrão pro agente principal
(`isHeartbeatEnabledForAgent`). Log confirma `[heartbeat] started` em todo
boot, incluindo às 22:22:53 do dia 15/07.

**Achado decisivo:** `before_agent_reply` só é invocado com
`trigger === "cron"` — nunca com `"heartbeat"`. Os dois únicos call sites
reais (`src/agents/embedded-agent-runner/run.ts:869`,
`src/agents/cli-runner.ts:405-408`) guardam com
`if (params.trigger === "cron" && ...)`. O check defensivo do próprio
`memory-core` pra `ctx.trigger === "heartbeat"` (`dreaming.ts:931`) é
código morto no harness atual — nunca alcançado.

**Pergunta 2 — sob que condição faz o trabalho pesado?** Só com
`trigger === "cron"` e um cron job de "managed dreaming" existente. Consulta
direta à SQLite compartilhada (`state/openclaw.sqlite`): tabela `cron_jobs`
tem **zero linhas**, `cron_run_logs` também **zero** — nenhum job desse
tipo jamais existiu ou rodou. Mesmo que existisse, o timer do `CronService`
nunca dispara nada com `cron.enabled: false` (`src/cron/service/
timer.ts:1071`, `ops.ts:219` — early-return antes de armar qualquer timer).

**Pergunta 3 — bate com o timing do incidente (22:22-22:34)?** Não pode
bater: "analisa o mox" foi uma mensagem direta do WhatsApp
(`trigger` normal, não `"cron"`), então o hook nem seria invocado nesse
turno. Mesmo um heartbeat/cron concorrente na mesma janela não ativaria o
ramo pesado, porque a pré-condição (job de dreaming existente) nunca foi
satisfeita neste deployment.

**Conclusão:** mesmo padrão de refutação do `acpx` — código genuinamente
perigoso em abstrato (a Retomada 7 estava certa sobre o hook não ter
timeout), mas inerte neste deployment específico porque a pré-condição de
disparo (job de dreaming via cron) nunca é satisfeita. Os dois suspeitos de
"perigo alto" da Retomada 7 caem por razões diferentes mas do mesmo tipo:
atribuição de causa por leitura de código plausível, sem confirmar contra o
estado real do sistema.

## Retomada — Disjuntores + Sub-agentes Paralelos, com regra empírica obrigatória

Mudança de método pedida explicitamente pelo usuário: combinar a técnica de
sub-agentes paralelos do Antigravity com a correção do erro que se repetiu
3 vezes (acpx, memory-core, e os 7 hooks restantes — todos "julgados" só por
leitura de código). Regra obrigatória desta rodada: nenhum sub-agente pode
concluir "achei a causa" sem reproduzir o travamento de verdade, com
evidência ao vivo.

### Ambiente isolado montado antes de lançar os sub-agentes

Pra permitir reprodução real sem tocar produção nem mandar mensagem real de
WhatsApp: stack Docker Compose isolado, projeto `bug4repro` (containers
`bug4repro-openclaw-gateway-1`, `bug4repro-openclaw-cli-1`), imagem
`openclaw:bug4repro` (buildada do worktree atual — já inclui o fix do
Antigravity + a instrumentação `debug-timing.ts` da sessão de 15/07), config
dir isolado em `/tmp/.../scratchpad/bug4-repro/openclaw-home` (nunca
`~/.openclaw` real), portas 28789/28790/13978 (só localhost). `openclaw.json`
isolado replica exatamente a config real do incidente (`plugins.entries`:
`deepseek` + `github-repo-report`, `tools.sandbox.tools.alsoAllow:
["github_repo_report"]`, `agents.defaults.sandbox.mode: "off"`), sem o canal
`whatsapp-cloud` — turnos disparados via `openclaw agent --message "..."`
(CLI direto, mesmo pipeline real de harness/hooks, zero canal envolvido, zero
risco de mensagem real).

**Nota de segurança sobre o build:** `docker compose build` sem `-p`/`--tag`
explícito escreve na MESMA tag `openclaw:local` usada pela produção real.
O container de produção já rodando não foi afetado (containers referenciam
o ID da imagem no momento da criação, não a tag), mas a tag `openclaw:local`
agora aponta pra imagem nova — se alguém rodar `docker compose up -d`
(recriar, não só restart) no projeto de produção sem perceber, vai pegar essa
imagem nova. Retaggeada para `openclaw:bug4repro` para uso exclusivo do teste,
mas o fato fica registrado aqui.

### Sub-agente A — reprodução ao vivo (resultado: NÃO reproduziu, com evidência real)

21 invocações reais de `openclaw agent`, 17 delas com o pipeline completo de
`github_repo_report` de fato disparando (sequencial, paralelo com sessões
diferentes, paralelo na mesma session key). Todas concluíram com sucesso,
mais lenta ~1.2s de execução interna da tool. Todas as 12 marcas de timing
(`workspace:create` → `http:start` → `headers-received` → `body-read` →
`workspace-write` → `extract` → `walk` → `inline-files` → `run-callback` →
`workspace:cleanup`) apareceram completas e em ordem, sem gap anormal.

**Achado real ao longo do caminho (verificado no código, não suposição):** as
3 primeiras tentativas com a mensagem literal "analisa o mox" nunca chegaram
a chamar a tool — não por onboarding incompleto (`setupCompletedAt` já
preenchido), mas porque `github-repo-report` tem `toolMetadata.optional: true`
no manifest, e `src/plugins/tools.ts`
(`pluginToolNamesMatchAllowlist`/`isOptionalToolEntryPotentiallyAllowed`)
exige que tools opcionais estejam no allowlist **top-level**
(`tools.alsoAllow`), não no `tools.sandbox.tools.alsoAllow` (que só rege
política de sandbox, irrelevante com `sandbox.mode: "off"`). Corrigido no
`openclaw.json` isolado (não é produção); a partir daí a tool passou a
disparar de verdade em toda tentativa seguinte.

**Conclusão do sub-agente A:** uma chamada isolada de `github_repo_report`
via harness real — mesmo repetida, mesmo em paralelo — não é suficiente pra
reproduzir o Bug 4 nas condições testadas (sem tráfego real do WhatsApp, sem
os outros 8 plugins gerando concorrência simultânea de verdade no processo
do gateway). Reforça a hipótese 2 da Retomada 4 (concorrência real do
processo, não a chamada isolada da tool).

### Sub-agente B — monitoramento contínuo (resultado: confirma independentemente que nada travou)

Script de monitoramento (`monitor.sh`, rodando como processo de shell
destacado, independente do próprio turno do agente) amostrou `docker stats`
+ `/proc/<pid>/status` a cada ~2-3s por toda a janela das tentativas do
sub-agente A (~13 minutos, 253 amostras salvas em
`scratchpad/bug4-repro/monitor.log`). CPU nunca caiu a zero absoluto (mínimo
observado ~0.4-0.5%, consistente com overhead normal de idle/healthcheck),
sem nenhuma janela de 20s+ com CPU+rede zeradas — o padrão que caracterizaria
uma repetição do Bug 4. Picos de CPU (até 178%) e memória coincidem com os
horários reais das chamadas do sub-agente A, confirmando que o monitor
capturou atividade real, não um processo ocioso sem trabalho nenhum.

### Sub-agente C — varredura dos 7 hooks restantes (resultado: todos descartados)

Aplicou a mesma cadeia de verificação já validada pro acpx/memory-core em
cada um dos 7 hooks sem timeout default que sobraram da Retomada 7
(`agent_turn_prepare`, `message_sending`, `message_sent`, `message_received`,
`before_dispatch`, `llm_input`, `llm_output`): (1) confirma ausência de
timeout no código atual, (2) grep por `api.on("<hook>"` em toda
`extensions/`, (3) cruza contra a lista real de plugins carregados neste
deployment (`browser, canvas, deepseek, device-pair, file-transfer,
github-repo-report, memory-core, phone-control, talk-voice` + canal
`whatsapp-cloud`).

Resultado: **todos os 7 descartados** — o único registro real de cada hook
vem de um plugin que não está carregado aqui (`thread-ownership`, específico
de Slack, registra `message_sending`/`message_received` mas não está
instalado) ou não tem nenhum registro real em produção (`agent_turn_prepare`
só em fixture de teste; `message_sent`, `before_dispatch`, `llm_input`,
`llm_output` sem nenhum `api.on(...)` real em lugar nenhum do repo). Todos os
call sites reais confirmados como no-op via `hasHooks(...)` gates.

**Cruzamento com o log de tentativas do sub-agente A:** o relatório final do
B (chegou em duas partes — a primeira notificação foi só um "aguardando",
a segunda trouxe a análise completa) apontou 10 janelas de 20s+ com CPU baixa
e rede estática, as duas maiores (~52.8s e ~50.8s) dentro do MESMO container
(sem recriação). Cruzando com os timestamps exatos de `trigger.log`: a
primeira janela (09:59:35-10:00:27 UTC) cobre o intervalo ocioso entre o fim
da tentativa 2 (09:58:08) e o início da tentativa 3 (10:00:22), mais uma
sobreposição mínima com a própria tentativa 3 — que é justamente uma das 3
tentativas que NUNCA chamaram a tool (bug do `tools.alsoAllow` ainda não
corrigido), então CPU baixa ali é esperado, não suspeito. A segunda janela
(10:01:05-10:01:56 UTC) cai inteiramente DEPOIS da tentativa 3 e ANTES da
nota "ENV FIX" (10:02:53) — período em que o sub-agente A estava lendo
código/ajustando config, não chamando nada. Nenhuma das janelas coincide com
uma tentativa realmente em andamento (todas as 17 chamadas reais da tool
completaram em segundos, nunca 20s+). Os "2 eventos de recriação de
container" que o B também notou (`docker exec`/`docker stats` falhando por
completo, seguido de novo Container ID) batem com os restarts do gateway
que o A fez pra recarregar o `openclaw.json` corrigido — operação legítima
no ambiente isolado, não uma trava. Conclusão: nenhuma evidência de B
contradiz o "não reproduziu" do A.

Isso **esgota** a lista completa de 9 hooks sem timeout apontados na
Retomada 7: 2 já descartados antes (`before_agent_reply`/memory-core,
`reply_dispatch`/acpx), 1 já corrigido (`reply_payload_sending`, commit
`12c785e9bf`), e os 7 acima agora descartados por ausência de handler
registrado. **Se o Bug 4 é causado por um hook de plugin desprotegido, essa
lista já não tem mais candidato — o vetor "hook sem timeout + plugin ativo"
está esgotado.**

### Conclusão desta retomada

**Bug 4 continua sem causa raiz confirmada.** Mas o espaço de busca ficou
bem mais estreito e com evidência real, não especulação: nenhum dos 9 hooks
desprotegidos do framework explica o incidente (plugin não carregado, trigger
nunca ocorre, ou handler inexistente), e uma reprodução direta via harness
real (bypassando WhatsApp) — incluindo variações com paralelismo — não
disparou o travamento. Isso aponta com mais força pra hipótese 2 da Retomada
4: contenção de recurso real (webhook HTTP + exec de sandbox Docker + SQLite
compartilhada, todos ativos ao mesmo tempo) que só aparece sob carga
concorrente genuína do gateway de produção, não reproduzível num processo
isolado fazendo só essa chamada — mesmo com múltiplas chamadas paralelas
simuladas.

  disparar) ou lógica fora do sistema de hooks inteiramente.
