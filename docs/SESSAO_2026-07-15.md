## Investigação: poder específico (não genérico) — Amigão + análise de repositórios GitHub

Investigação de código-fonte real do OpenClaw (não documentação
genérica) para desenhar uma capacidade nova e restrita: o Amigão
clonar/ler UM repositório GitHub específico e gerar relatório,
sem abrir exec genérico nem mexer nas camadas de segurança
existentes. Todas as citações são `caminho/dentro/de/openclaw/:linha`,
verificadas lendo o código, não assumidas.

### Correção importante: "skill" no OpenClaw ≠ "skill" no Claude Code

São dois conceitos diferentes no OpenClaw:

| Conceito | Nome real | O que é | O que controla |
|---|---|---|---|
| Função que o modelo chama | **`tool`** (`AnyAgentTool`, `src/agents/tools/common.ts:26-38`) | Objeto TS `{name, description, parameters, execute}` | Gatilhado por `tools.allow/deny/profile` |
| Arquivo `SKILL.md` | **`skill`** | Só instrui o modelo *como* usar tools já existentes | Gatilhado por `agents.defaults.skills`, filtrado em `src/skills/discovery/agent-filter.ts:25-37` |

**Achado:** `agents.defaults.skills: ["exec"]` no `openclaw.json` atual
provavelmente **não restringe tool nenhuma** (não existe
`skills/exec/SKILL.md` no repo — esse campo só filtra instruções
de prompt). A restrição real de tools no sandbox vem de outro
lugar: `DEFAULT_TOOL_ALLOW = [exec, process, read, write, edit,
apply_patch, image, sessions_*, subagents, session_status]`
(`src/agents/sandbox/constants.ts:18-33`), aplicado em
`src/agents/sandbox/tool-policy.ts:217-268`. A proteção continua
valendo — só que por um mecanismo diferente do documentado em
`ARQUITETURA_SEGURANCA.md`.

### `web_fetch` roda fora do sandbox

Tool sem campo de comando livre (`{url, extractMode?, maxChars?}`,
`src/agents/tools/web-fetch.ts:66-80`) executada no **processo do
gateway (host)**, sem relação com o Docker do sandbox — confirmado
em `extensions/firecrawl/src/firecrawl-fetch-provider.ts:12-32`
(função `execute` comum fazendo `fetch()` direto). Nem está na
allowlist padrão do sandbox. Isso é o precedente-chave: uma tool
nova pode buscar dados externos (GitHub) sem nunca entrar no
container isolado.

### `before_tool_call` / `registerTrustedToolPolicy`

Hook real (`src/plugins/hook-types.ts:95,142`), motor em
`src/agents/agent-tools.before-tool-call.ts` (1695 linhas). Bloqueia
a chamada específica (é `await`ado), pode `block`, mutar `params`
ou `requireApproval`. Template real de auto-allow condicional por
argumento: `src/skills/workshop/policy.ts:47-70` (core, não plugin).
Único exemplo real de **plugin** registrando essa política:
`src/plugins/contracts/host-hook-fixture.ts:63-73` (fixture de
teste, mas é o único wiring real de `registerTrustedToolPolicy`
via `OpenClawPluginApi` no repo).

### `before_agent_finalize` BLOQUEIA a resposta — `agent_end` não

Cadeia confirmada: `src/agents/embedded-agent-runner/run/attempt.ts:3504`
(`await runAgentHarnessBeforeAgentFinalizeHook`) →
`src/agents/harness/lifecycle-hook-helpers.ts:172` →
`src/plugins/hooks.ts:661` → trava
`deliverTerminalWithLifecycleErrorFallback`
(`src/agents/embedded-agent-subscribe.handlers.lifecycle.ts:365-383`,
a função que manda a mensagem pro WhatsApp). **Sem timeout padrão**
(`src/plugins/hooks.ts:222-233` não lista esse hook) — um agente de
auditoria pesado ali pode travar a resposta indefinidamente.

**Achado-chave:** `agent_end` (`src/plugins/hook-types.ts:367-373`)
é disparado **sem `await`** (`src/agents/harness/agent-end-side-effects.ts:34`),
depois da resposta já ter sido decidida, com timeout de 30s +
`unref` (não trava nem o processo). Exemplo real de plugin usando:
`extensions/memory-lancedb/index.ts:1943` (`api.on("agent_end", ...)`).
É o mecanismo certo pra auditoria que não pode atrasar o WhatsApp.

### Armazenamento: KV genérico vs tabela SQLite dedicada

A API padrão de plugin (`api.runtime.state.openKeyedStore`,
`src/plugin-sdk` → `src/plugins/registry.ts:2676-2685`) só oferece
blob JSON por chave (bom pra estado simples, ruim pra log
consultável com colunas). Existe precedente real de plugin bundled
com **banco SQLite próprio e colunas reais**:
`extensions/workboard/src/sqlite-store.ts:1-45` — abre
`node:sqlite` direto em `state/plugins/workboard/workboard.sqlite`
via `resolveStateDir()`, com schema/migração próprios. Esse é o
padrão certo pra tabela de auditoria com colunas (`repo`, `ref`,
`decision`, `timestamp`), não o KV genérico.

### Estrutura de plugin (templates reais)

Dois plugins bundled pequenos servem de modelo estrutural
(`extensions/diffs/`, `extensions/firecrawl/`): `package.json` +
`openclaw.plugin.json` (manifesto separado, campos `id`,
`activation.onStartup`, `contracts.tools`) na raiz, `index.ts`
(entry point, export default via `defineToolPlugin`), `api.ts` +
`runtime-api.ts` (barril público pra outros módulos importarem),
lógica real em `src/*.ts`, testes colocados (`*.test.ts` ao lado
do arquivo testado). Exemplo mínimo real de `defineToolPlugin`:
`extensions/llm-task/index.ts:1-31`.

**Plugin bundled não precisa de `plugins.allow`** — é descoberto
automaticamente por scan de diretório
(`src/plugins/discovery.ts:1518,1546,1565`, `origin: "bundled"`
excluído do aviso em `src/plugins/loader-provenance.ts:229-231`).
Mas a tool nova **fica bloqueada no sandbox por padrão** até ser
adicionada em `tools.sandbox.tools.alsoAllow` (aditivo, preferido)
ou `tools.sandbox.tools.allow` (substitui a lista) no `openclaw.json`
(`src/agents/sandbox/tool-policy.ts:217-268`).

## Implementação: `extensions/github-repo-report/` — 22 arquivos, 3 bugs reais encontrados e corrigidos

Plugin criado em `openclaw/extensions/github-repo-report/` seguindo
o desenho aprovado: tool `github_repo_report` com schema fechado

do gateway (nunca no sandbox), `registerTrustedToolPolicy` modelada
em `src/skills/workshop/policy.ts` (auto-allow pro repo habilitado,
`requireApproval` pro resto), auditoria via hook `agent_end`
gravando no KV do plugin. `src/repo-registry.ts` como fonte única
da verdade: só `Mox---Sistemas` com `enabled: true`; `meu-agente`
e `arbo` presentes no enum mas desabilitados.

Testado localmente antes de qualquer deploy: `pnpm test
extensions/github-repo-report` (24 testes) e `pnpm tsgo:extensions`
(typecheck do monorepo inteiro) — ambos limpos. Conectado ao
`openclaw.json` (plugin habilitado + `tools.sandbox.tools.alsoAllow`)
só depois de revisão manual do código (`repo-registry.ts` e
`policy.ts` colados linha a linha pro usuário conferir).

### Bug 1 — `configSchema` ausente derrubou o gateway em produção

`openclaw.plugin.json` sem o campo `configSchema` (achei que era
opcional pra um plugin sem config — não é). Rebuild #1 + deploy →
gateway entrou em **crash loop real**, WhatsApp caiu:
```
Gateway failed to start: Invalid config at openclaw.json.
plugins: plugin: plugin manifest requires configSchema
```
Recuperação: rollback imediato pra imagem Docker anterior (dangling,
24h, confirmada boa) — downtime real de ~1min. Fix: adicionar
`configSchema: {type: "object", properties: {}, additionalProperties:
false}`. Validado localmente com `pnpm dev` (sem Docker) antes do
rebuild #2, confirmando "9 plugins" incluindo `github-repo-report`
sem erro — só depois disso rebuildou de verdade.

### Bug 2 — `ref: undefined` quebrou a gravação da auditoria

`policy.ts` sempre define a chave `ref` (string ou `undefined`
explícito) ao registrar o pedido pendente. O KV store do plugin
exige valor inteiro JSON-serializável e rejeita `undefined`
explícito em qualquer campo:
```
[hooks] agent_end handler from github-repo-report failed:
plugin state value at value.ref must be JSON-serializable
```
Confirmado com prova direta no banco (`SELECT COUNT(*) FROM
plugin_state_entries` = 0) — a chamada funcionou (o Amigão buscou
o repo real e respondeu bem no WhatsApp, confirmado pelo usuário),
só a auditoria falhava silenciosamente, sem travar a resposta
(confirma que o design "assíncrono, nunca bloqueia" aguentou até
sob falha real). Fix: `stripUndefinedValues()` remove qualquer
campo `undefined` antes de gravar — não só `ref`, qualquer campo
(`sessionKey`, `agentId`, `runError` etc). Coberto por teste novo
em `audit-log.test.ts` reproduzindo o caso real (`ref: undefined`
explícito, igual o `policy.ts` manda) antes de rebuildar de novo.

### Bug 3 — timeout não cobria a leitura do corpo HTTP, travou uma conversa

Em `github-fetch.ts`, o `clearTimeout` do `AbortController` rodava
logo após os cabeçalhos da resposta chegarem, **antes** de ler o
corpo (`response.arrayBuffer()`) — deixando a leitura do corpo sem
nenhuma proteção de timeout. No reteste pós-fix do Bug 2, a
conversa travou de verdade: 4+ minutos sem atividade, CPU ociosa,
tráfego de rede **zerado e parado** (confirmado com `docker stats`
duas vezes, 8s de intervalo, bytes idênticos — não era download
lento, era conexão travada). Recuperação: `docker compose restart`
simples resolveu (sem precisar trocar imagem). Fix identificado
(mover `clearTimeout` pro `finally` envolvendo a função inteira,
não só o `fetch()` inicial) — **não aplicado/rebuildado ainda**,
fica pendente pra próxima sessão.

## Retomada 2 — `docker compose build` confirmado limpo, regressão do Docker CLI, Bug 4 novo (deadlock), tool revertida de novo

### `docker compose build` valida a pendência de ambiente da retomada anterior

Rodado `docker compose build` (só build, sem down/up) pra confirmar
se o build real (via Dockerfile, Node oficial `node:24-bookworm`
dentro do container) é afetado pela limitação do Node do host. Não
é: `pnpm build:docker` completou sem erro dentro do container, imagem
`openclaw:local` gerada com sucesso. Confirmado por extração direta
do arquivo de dentro da imagem (`docker run --rm --entrypoint cat
openclaw:local /app/extensions/github-repo-report/src/github-fetch.ts`)
que o fix do Bug 3 estava presente. A pendência de ambiente documentada
na retomada anterior fica resolvida: builds locais via `pnpm build`
direto no host não funcionam (Node do Debian sem suporte a TS), mas
`docker compose build`/`docker build` funcionam normalmente porque
rodam dentro do container com Node oficial.

### Regressão real causada pelo build: Docker CLI ausente na imagem nova

Antes de reconectar a tool, comparei a imagem nova com a que estava
rodando (dangling, ~11h, confirmada boa): a imagem anterior tinha
`docker` CLI em `/usr/bin/docker`; a nova buildada com `docker compose
build` puro, não. Causa: `ARG OPENCLAW_INSTALL_DOCKER_CLI=""` no
Dockerfile só vira `1` se passado explicitamente via `--build-arg`;
`scripts/docker/setup.sh:646-647,763` seta isso automaticamente
quando `OPENCLAW_SANDBOX` está ativo, mas nem `docker-compose.yml`
nem `.env` guardam esse argumento — quem buildou a imagem anterior
(alguma sessão passada) deve ter usado `setup.sh` ou passado o
`--build-arg` manualmente, e isso nunca ficou registrado em nenhum
arquivo do repo. `docker compose build` sozinho não tem como saber
disso.

**Impacto real, confirmado ao vivo:** com `agents.defaults.sandbox.mode:
"all"` exigindo o `docker` CLI dentro do container pra gerenciar
sandbox por `/var/run/docker.sock`, toda resposta real do agente
(inclusive um simples "oi") começava a falhar:
```
lane task error: ... "docker" command was not found in PATH
Embedded agent failed before reply: Sandbox mode requires Docker...
```
Webhook respondia 200 pra Meta (sem retry storm), mas nenhuma resposta
chegava no WhatsApp.

**Recuperação:** rollback imediato — `docker tag <imagem-anterior-dangling>
openclaw:local` + `docker compose down && up`. Confirmado `docker` CLI
de volta e "oi" respondendo normalmente (sem o fix do Bug 3 nessa
imagem, mas WhatsApp estável).

**Fix definitivo:** build direto (sem passar pelo `scripts/docker/setup.sh`,
que faz onboarding interativo, reescreve `gateway.mode/bind/allowedOrigins`,
muda `agents.defaults.sandbox.mode` pra `"non-main"` ou `"off"`, e pode
gerar um `OPENCLAW_GATEWAY_TOKEN` novo — efeitos colaterais grandes demais
pra esse objetivo) com só o build-arg necessário:
```
docker build --build-arg OPENCLAW_INSTALL_DOCKER_CLI=1 -t openclaw:local -f Dockerfile .
```
Confirmado depois do build: `docker` CLI presente **e** fix do Bug 3
presente na mesma imagem. `openclaw.json` e `.env` preservados
intactos (nenhum efeito colateral do `setup.sh`).

### Bug 4 — deadlock real no turno do `github_repo_report`, sem nenhum sinal de rede ou CPU

Depois do rebuild com Docker CLI + fix do Bug 3, reconectado o plugin
no `openclaw.json` (mesmo diff de sempre: `plugins.entries.github-repo-report`
+ `tools.sandbox.tools.alsoAllow: ["github_repo_report"]`). `docker compose
down/up`, `healthz` OK, 9 plugins carregados sem erro, teste "oi" limpo
(2 blocos de resposta, sem erro de sandbox — Docker CLI confirmado
funcionando).

Teste "analisa o mox" simulado em seguida: turno iniciou normalmente
(`WhatsApp Cloud reply started`, tool-policy resolvida, mesmo `traceId`
em dois blocos de resposta às 22:23:28 e 22:23:34) e então **travou
completamente** — 3+ minutos sem nenhuma atividade:
- `plugin_state_entries` (tabela de auditoria) permaneceu **vazia**
  (`agent_end` nunca disparou)
- `docker stats` do container gateway: CPU ociosa (~0.5-0.7%), NET I/O
  **idêntico** em duas leituras (sem atividade)
- `docker stats` do container de sandbox persistente
  (`openclaw-sbx-agent-main-*`): também 0% CPU, 0B/0B rede
- `/proc/net/tcp` dentro do gateway: nenhuma conexão `ESTABLISHED`
  (só `LISTEN`/`TIME_WAIT`) — nenhuma requisição de rede em voo, nem
  pro GitHub, nem pro DeepSeek

**Diferença chave em relação ao Bug 3:** o Bug 3 original tinha uma
conexão TCP **viva e travada** (bytes parados mas socket aberto,
confirmado por duas leituras de `docker stats` 8s de intervalo com
tráfego zerado). O Bug 4 não tem **nenhuma** atividade de rede ou CPU
em lugar nenhum do sistema (nem gateway, nem sandbox) — indica um
deadlock em algum `await` que nunca resolve dentro do processo Node,
não um problema de timeout de rede. Já mandou 2 blocos de resposta
parciais pro WhatsApp antes de travar, então não é uma falha
imediata — trava no meio do processamento.

**Causa raiz: não investigada ainda.** Hipóteses não descartadas:
promise pendente sem timeout em algum ponto do harness do agente
(padrão já visto antes nesta sessão: `before_agent_finalize` não tem
timeout padrão, `src/plugins/hooks.ts:222-233`), interação entre
`registerTrustedToolPolicy` e o dispatcher de resposta em buffer, ou
algo específico do `report-builder.ts`/`repo-registry.ts` não coberto
pelos testes unitários existentes. Precisa de investigação dedicada
(thread dump / `node --prof` / logging mais granular) numa próxima
sessão — não tentado hoje por decisão consciente de não seguir

## Retomada 3 — hipótese do DeepSeek pro Bug 4 avaliada e descartada, instrumentação adicionada

Antes de tentar qualquer fix pro Bug 4, avaliada a hipótese trazida
pelo DeepSeek: que `tar.x()` (extração do tarball, via `@openclaw/
fs-safe/archive`) usa zlib nativo pra descompressão gzip, e que o
`AbortSignal` passado pra ele não interrompe esse código nativo se o
tarball vier truncado — explicando o CPU/rede zerados observados no
Bug 4. Fix proposto: envolver `tar.x()` num `Promise.race` com
`setTimeout` puro, independente de o sinal chegar no código nativo.

**Verificação ao vivo (conforme pedido, antes de aplicar):** `docker
exec` nos containers `openclaw-openclaw-gateway-1` e
`openclaw-openclaw-cli-1` checando `/tmp/github-repo-report-*/
extracted/` — nenhum diretório de workspace existe em nenhum dos
dois containers agora (a tool está desconectada desde o fim da
retomada anterior, sem turno em andamento). Não deu pra confirmar a
hipótese via extração parcial/vazia ao vivo por falta de repro ativo
no momento.

**Leitura do código-fonte real da dependência (`node_modules/
@openclaw/fs-safe/dist/archive.js` + `archive-deadline.js`) refuta a
hipótese como está.** `extractArchive()` já envolve toda a chamada
de `tar.x()` — via `withExtractionDeadline()` → `waitForDeadline()`
— num `Promise.race` idêntico ao proposto: uma promise que resolve
o trabalho real (`tar.x()` incluído) contra uma segunda promise que
rejeita quando o `AbortSignal` dispara, e esse disparo é feito por
um `setTimeout` JavaScript puro (`archive-deadline.js:43-45`),
totalmente independente de o código nativo do zlib responder ao
sinal. Ou seja: a proteção "desiste depois do timeout não importa o
que rolar por baixo" que o fix do DeepSeek propõe **já existe**, uma
camada abaixo de onde o fix seria aplicado (dentro de
`extractArchive`, não em volta dela). Aplicar o mesmo padrão de novo
em `github-fetch.ts`, por cima de uma chamada que já tem esse exato
mecanismo por dentro, seria redundante — não muda o comportamento no
cenário que o DeepSeek descreve.

**Por que isso não bate com o Bug 4 observado:** o travamento durou
3+ minutos sem nenhuma recuperação, exigindo `docker compose
restart`. Se o mecanismo de timeout de 30s (`EXTRACT_TIMEOUT_MS`) já
presente em `extractArchive` estivesse funcionando — mesmo que o
`tar.x()` nativo ignorasse o sinal —, o `await extractArchive(...)`
em `fetchGithubRepoTarball` deveria ter rejeitado sozinho por volta
de 30s, e o turno teria terminado (com erro), não travado
indefinidamente. Precisar de restart do container é mais consistente
com o próprio event loop do Node tendo travado (uma chamada nativa
verdadeiramente síncrona e bloqueante), cenário em que nenhum
`Promise.race` em JavaScript — nem o que já existe, nem uma cópia
dele — consegue resgatar o processo, porque o callback do
`setTimeout` também depende do event loop estar vivo pra disparar.
Também pesa contra a hipótese: a sessão registrou **zero atividade
de rede durante toda a janela de 3+ minutos**, nem uma conexão
efêmera — um pouco inconsistente com "baixou um tarball truncado e
travou descomprimindo".

`report-builder.ts` (passo pós-extração, caminhada local limitada a
`MAX_TREE_ENTRIES`) e o cleanup do `withTempWorkspace` (um `fs.rm`
local simples) também foram lidos e não mostram nenhum ponto óbvio
de bloqueio sem timeout.

**Decisão:** não aplicado o fix do DeepSeek — não haveria evidência
de que resolveria o travamento real, e aplicá-lo documentado como
"Bug 4 corrigido" arriscaria reconectar a tool no WhatsApp de
produção com uma falsa sensação de segurança.

### Instrumentação adicionada (sem reconectar a tool)

Adicionado logging com timestamp relativo (`performance.now()`) em
cada fronteira de `await` da tool, via um novo helper local
`extensions/github-repo-report/src/debug-timing.ts`
(`createTimingMarker`), roteado por `PluginLogger` (`api.logger`,
já exportado por outras extensions como `diffs`) através de
`plugin.ts` → `tool.ts` → `github-fetch.ts` / `report-builder.ts`:

- `tool.ts`: `workspace:create:start/done`, `run-callback:done`,
  `workspace:cleanup:done` (esse último isola o `fs.rm` de cleanup
  do `withTempWorkspace`, que não dá pra instrumentar por dentro por
  viver em `@openclaw/fs-safe`, dependência externa).
- `github-fetch.ts`: `http:start`, `http:headers-received`,
  `body-read:start/done`, `workspace-write:start/done`,
  `extract:start/done`.
- `report-builder.ts`: `walk:start/done`,
  `inline-files:start/done`.

Cada mark é `logger?.debug?.(...)` — silencioso em produção salvo
nível de log `debug`, sem custo de I/O real quando desligado. Não
remove nenhuma proteção existente, não muda comportamento de
runtime, só adiciona visibilidade. `pnpm test extensions/
github-repo-report` (25 testes) e `pnpm tsgo:extensions` (código de
produção) passam limpos com a instrumentação. `pnpm tsgo:
extensions:test` acusa erros pré-existentes de tipo em três arquivos
de teste não tocados nesta retomada (`audit-log.test.ts`,
`schema.test.ts`, `tool.test.ts` — acesso a `.properties`/`.enum`
em tipos `TSchema`/`TUnsafe`); não são causados por essa mudança e
ficam registrados aqui pra não se perderem, mas não foram
investigados/corrigidos agora por estarem fora do escopo do Bug 4.

**Objetivo:** na próxima reprodução do travamento (local, sem
tráfego real do WhatsApp — decisão consciente de não repetir o teste
em produção), os logs devem apontar exatamente qual `await` nunca
retorna, permitindo diagnosticar se é de fato um event-loop stall
nativo (nenhum mark novo aparece depois de um ponto, sem erro) ou
uma pendência sem timeout em outro lugar do harness. Reprodução
fica para quando houver decisão consciente de rodar o teste — não
feita nesta retomada.

**Estado:** `openclaw.json` permanece revertido (tool desconectada,
sem `plugins.entries.github-repo-report`, sem
`tools.sandbox.tools.alsoAllow`), igual ao fim da retomada anterior.
Nenhuma mudança de runtime/deploy nesta retomada — só código de
instrumentação, ainda não implantado em nenhuma imagem Docker.
