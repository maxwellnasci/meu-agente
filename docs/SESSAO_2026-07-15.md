+ assinatura HMAC válida → resultado: bloqueado

está funcionando como projetado.

## Investigação: poder específico (não genérico) — Amigão + análise de repositórios GitHub

Investigação de código-fonte real do OpenClaw (não documentação
genérica) para desenhar uma capacidade nova e restrita: o Amigão
clonar/ler UM repositório GitHub específico e gerar relatório,
sem abrir exec genérico nem mexer nas camadas de segurança
existentes. Todas as citações são `caminho/dentro/de/openclaw/:linha`,
verificadas lendo o código, não assumidas.

### Correção importante: "skill" no OpenClaw ≠ "skill" no Claude Code

São dois conceitos diferentes no OpenClaw:

| Conceito | Nome real | O que é | O que controla |
|---|---|---|---|
| Função que o modelo chama | **`tool`** (`AnyAgentTool`, `src/agents/tools/common.ts:26-38`) | Objeto TS `{name, description, parameters, execute}` | Gatilhado por `tools.allow/deny/profile` |
| Arquivo `SKILL.md` | **`skill`** | Só instrui o modelo *como* usar tools já existentes | Gatilhado por `agents.defaults.skills`, filtrado em `src/skills/discovery/agent-filter.ts:25-37` |

**Achado:** `agents.defaults.skills: ["exec"]` no `openclaw.json` atual
provavelmente **não restringe tool nenhuma** (não existe
`skills/exec/SKILL.md` no repo — esse campo só filtra instruções
de prompt). A restrição real de tools no sandbox vem de outro
lugar: `DEFAULT_TOOL_ALLOW = [exec, process, read, write, edit,
apply_patch, image, sessions_*, subagents, session_status]`
(`src/agents/sandbox/constants.ts:18-33`), aplicado em
`src/agents/sandbox/tool-policy.ts:217-268`. A proteção continua
valendo — só que por um mecanismo diferente do documentado em
`ARQUITETURA_SEGURANCA.md`.

### `web_fetch` roda fora do sandbox

Tool sem campo de comando livre (`{url, extractMode?, maxChars?}`,
`src/agents/tools/web-fetch.ts:66-80`) executada no **processo do
gateway (host)**, sem relação com o Docker do sandbox — confirmado
em `extensions/firecrawl/src/firecrawl-fetch-provider.ts:12-32`
(função `execute` comum fazendo `fetch()` direto). Nem está na
allowlist padrão do sandbox. Isso é o precedente-chave: uma tool
nova pode buscar dados externos (GitHub) sem nunca entrar no
container isolado.

### `before_tool_call` / `registerTrustedToolPolicy`

Hook real (`src/plugins/hook-types.ts:95,142`), motor em
`src/agents/agent-tools.before-tool-call.ts` (1695 linhas). Bloqueia
a chamada específica (é `await`ado), pode `block`, mutar `params`
ou `requireApproval`. Template real de auto-allow condicional por
argumento: `src/skills/workshop/policy.ts:47-70` (core, não plugin).
Único exemplo real de **plugin** registrando essa política:
`src/plugins/contracts/host-hook-fixture.ts:63-73` (fixture de
teste, mas é o único wiring real de `registerTrustedToolPolicy`
via `OpenClawPluginApi` no repo).

### `before_agent_finalize` BLOQUEIA a resposta — `agent_end` não

Cadeia confirmada: `src/agents/embedded-agent-runner/run/attempt.ts:3504`
(`await runAgentHarnessBeforeAgentFinalizeHook`) →
`src/agents/harness/lifecycle-hook-helpers.ts:172` →
`src/plugins/hooks.ts:661` → trava
`deliverTerminalWithLifecycleErrorFallback`
(`src/agents/embedded-agent-subscribe.handlers.lifecycle.ts:365-383`,
a função que manda a mensagem pro WhatsApp). **Sem timeout padrão**
(`src/plugins/hooks.ts:222-233` não lista esse hook) — um agente de
auditoria pesado ali pode travar a resposta indefinidamente.

**Achado-chave:** `agent_end` (`src/plugins/hook-types.ts:367-373`)
é disparado **sem `await`** (`src/agents/harness/agent-end-side-effects.ts:34`),
depois da resposta já ter sido decidida, com timeout de 30s +
`unref` (não trava nem o processo). Exemplo real de plugin usando:
`extensions/memory-lancedb/index.ts:1943` (`api.on("agent_end", ...)`).
É o mecanismo certo pra auditoria que não pode atrasar o WhatsApp.

### Armazenamento: KV genérico vs tabela SQLite dedicada

A API padrão de plugin (`api.runtime.state.openKeyedStore`,
`src/plugin-sdk` → `src/plugins/registry.ts:2676-2685`) só oferece
blob JSON por chave (bom pra estado simples, ruim pra log
consultável com colunas). Existe precedente real de plugin bundled
com **banco SQLite próprio e colunas reais**:
`extensions/workboard/src/sqlite-store.ts:1-45` — abre
`node:sqlite` direto em `state/plugins/workboard/workboard.sqlite`
via `resolveStateDir()`, com schema/migração próprios. Esse é o
padrão certo pra tabela de auditoria com colunas (`repo`, `ref`,
`decision`, `timestamp`), não o KV genérico.

### Estrutura de plugin (templates reais)

Dois plugins bundled pequenos servem de modelo estrutural
(`extensions/diffs/`, `extensions/firecrawl/`): `package.json` +
`openclaw.plugin.json` (manifesto separado, campos `id`,
`activation.onStartup`, `contracts.tools`) na raiz, `index.ts`
(entry point, export default via `defineToolPlugin`), `api.ts` +
`runtime-api.ts` (barril público pra outros módulos importarem),
lógica real em `src/*.ts`, testes colocados (`*.test.ts` ao lado
do arquivo testado). Exemplo mínimo real de `defineToolPlugin`:
`extensions/llm-task/index.ts:1-31`.

**Plugin bundled não precisa de `plugins.allow`** — é descoberto
automaticamente por scan de diretório
(`src/plugins/discovery.ts:1518,1546,1565`, `origin: "bundled"`
excluído do aviso em `src/plugins/loader-provenance.ts:229-231`).
Mas a tool nova **fica bloqueada no sandbox por padrão** até ser
adicionada em `tools.sandbox.tools.alsoAllow` (aditivo, preferido)
ou `tools.sandbox.tools.allow` (substitui a lista) no `openclaw.json`
(`src/agents/sandbox/tool-policy.ts:217-268`).

## Implementação: `extensions/github-repo-report/` — 22 arquivos, 3 bugs reais encontrados e corrigidos

Plugin criado em `openclaw/extensions/github-repo-report/` seguindo
o desenho aprovado: tool `github_repo_report` com schema fechado
(`{repo: enum, ref?: string}`), `execute` rodando 100% no processo
do gateway (nunca no sandbox), `registerTrustedToolPolicy` modelada
em `src/skills/workshop/policy.ts` (auto-allow pro repo habilitado,
`requireApproval` pro resto), auditoria via hook `agent_end`
gravando no KV do plugin. `src/repo-registry.ts` como fonte única
da verdade: só `Mox---Sistemas` com `enabled: true`; `meu-agente`
e `arbo` presentes no enum mas desabilitados.

Testado localmente antes de qualquer deploy: `pnpm test
extensions/github-repo-report` (24 testes) e `pnpm tsgo:extensions`
(typecheck do monorepo inteiro) — ambos limpos. Conectado ao
`openclaw.json` (plugin habilitado + `tools.sandbox.tools.alsoAllow`)
só depois de revisão manual do código (`repo-registry.ts` e
`policy.ts` colados linha a linha pro usuário conferir).

### Bug 1 — `configSchema` ausente derrubou o gateway em produção

`openclaw.plugin.json` sem o campo `configSchema` (achei que era
opcional pra um plugin sem config — não é). Rebuild #1 + deploy →
gateway entrou em **crash loop real**, WhatsApp caiu:
```
Gateway failed to start: Invalid config at openclaw.json.
plugins: plugin: plugin manifest requires configSchema
```
Recuperação: rollback imediato pra imagem Docker anterior (dangling,
24h, confirmada boa) — downtime real de ~1min. Fix: adicionar
`configSchema: {type: "object", properties: {}, additionalProperties:
false}`. Validado localmente com `pnpm dev` (sem Docker) antes do
rebuild #2, confirmando "9 plugins" incluindo `github-repo-report`
sem erro — só depois disso rebuildou de verdade.

### Bug 2 — `ref: undefined` quebrou a gravação da auditoria

`policy.ts` sempre define a chave `ref` (string ou `undefined`
explícito) ao registrar o pedido pendente. O KV store do plugin
exige valor inteiro JSON-serializável e rejeita `undefined`
explícito em qualquer campo:
```
[hooks] agent_end handler from github-repo-report failed:
plugin state value at value.ref must be JSON-serializable
```
Confirmado com prova direta no banco (`SELECT COUNT(*) FROM
plugin_state_entries` = 0) — a chamada funcionou (o Amigão buscou
o repo real e respondeu bem no WhatsApp, confirmado pelo usuário),
só a auditoria falhava silenciosamente, sem travar a resposta
(confirma que o design "assíncrono, nunca bloqueia" aguentou até
sob falha real). Fix: `stripUndefinedValues()` remove qualquer
campo `undefined` antes de gravar — não só `ref`, qualquer campo
(`sessionKey`, `agentId`, `runError` etc). Coberto por teste novo
em `audit-log.test.ts` reproduzindo o caso real (`ref: undefined`
explícito, igual o `policy.ts` manda) antes de rebuildar de novo.

### Bug 3 — timeout não cobria a leitura do corpo HTTP, travou uma conversa

Em `github-fetch.ts`, o `clearTimeout` do `AbortController` rodava
logo após os cabeçalhos da resposta chegarem, **antes** de ler o
corpo (`response.arrayBuffer()`) — deixando a leitura do corpo sem
nenhuma proteção de timeout. No reteste pós-fix do Bug 2, a
conversa travou de verdade: 4+ minutos sem atividade, CPU ociosa,
tráfego de rede **zerado e parado** (confirmado com `docker stats`
duas vezes, 8s de intervalo, bytes idênticos — não era download
lento, era conexão travada). Recuperação: `docker compose restart`
simples resolveu (sem precisar trocar imagem). Fix identificado
(mover `clearTimeout` pro `finally` envolvendo a função inteira,
não só o `fetch()` inicial) — **não aplicado/rebuildado ainda**,
fica pendente pra próxima sessão.
