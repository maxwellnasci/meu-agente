## Problemas encontrados e soluções

### 1. `gateway.mode` obrigatório no openclaw.json
**Sintoma:** Exit Code 78 ao tentar subir o container pela primeira vez.  
**Causa raiz:** o validador de esquema do OpenClaw exige `gateway.mode` explícito para prevenir configurações corrompidas ou infiltradas. O campo estava ausente no nosso openclaw.json inicial.  
**Solução:** adicionado `"gateway": { "mode": "local" }` no topo do json, com confirmação via documentação oficial (`docs.openclaw.ai/gateway/configuration-reference`).  
**Lição:** o OpenClaw usa "fail-closed" — prefere parar do que rodar com config ambígua. Isso é segurança intencional.

---

### 2. "Missing workspace template: AGENTS.md"
**Sintoma:** erro ao iniciar conversa na UI, gateway subia mas agente falhava na primeira interação.  
**Causa raiz:** durante a limpeza de ~790 arquivos feita anteriormente, os arquivos `docs/reference/templates/*.md` foram deletados fisicamente do diretório `openclaw/`. Como esses arquivos são idênticos nos commits nightly e v2026.6.9, o `git checkout` não os restaurou — eles continuaram ausentes no disco, mas o `git status` mostrava ` D` (deleted in working tree). O runtime exige `src/agents/templates/AGENTS.md` dentro da imagem compilada.  
**Solução:**
```bash
git -C openclaw/ restore docs/reference/templates/
```
Isso restaurou 12 arquivos do histórico git sem precisar de nada externo. A imagem foi reconstruída após o restore.

**Lição:** `git checkout <tag>` não restaura arquivos manualmente deletados se o conteúdo for o mesmo nos dois commits. Usar `git restore` explicitamente. Nunca deletar manualmente arquivos de um repo de terceiro sem entender o que o build usa.

### 4. "Unknown model: deepseek/deepseek-v4-flash" e "deepseek/deepseek-chat"
**Sintoma:** mesmo com modelo listado pelo `openclaw models list`, o runtime retornava `FailoverError: Unknown model` com `reason=model_not_found`.  
**Causa raiz:** o plugin registry (SQLite em `~/.openclaw/state/openclaw.sqlite`) estava ausente/stale. Sem o registry, o runtime não sabia que o provider DeepSeek estava habilitado, mesmo com a chave de API presente no ambiente.  
**Por que `openclaw doctor --fix` funcionou:** o comando `doctor --fix` é o "corretor automático" oficial do OpenClaw. Ele detecta providers com chaves de API configuradas mas sem plugin instalado, instala os plugins necessários, reconstrói o registry SQLite e corrige permissões de diretórios de estado. É o procedimento padrão pós-instalação/pós-troca-de-versão.  
**Solução:**
```bash
docker compose -f openclaw/docker-compose.yml exec openclaw-gateway openclaw doctor --fix
```
O `doctor --fix` fez automaticamente:
- Instalou o plugin: `@openclaw/deepseek-provider`
- Configurou o modelo: `deepseek/deepseek-chat enabled automatically` (V3, não V4)
- Reconstruiu o registry: `54/78 plugins indexed`
- Adicionou ao `openclaw.json`: `plugins.entries.deepseek.enabled: true`
- Corrigiu permissões do state dir para `700`

**Lição:** sempre rodar `openclaw doctor --fix` após primeira instalação ou troca de versão. O doctor é o mecanismo oficial de bootstrap — não improvisar com configurações manuais de provider/plugin.

## ⚠️ Estado de segurança honesto

**O agente está rodando SOLTO.** `sandbox.mode: "off"` significa que nenhum contêiner de isolamento é criado para execução de código. O agente executa comandos diretamente no processo do gateway, sem sandboxing.

**O que mitiga (parcialmente):**
- `workspaceAccess: "none"` — agente não tem volume com arquivos do host
- Sem skills habilitadas — o agente não tem ferramentas de execução no momento
- Container roda como usuário `node` (não root) — sem privilege escalation fácil

**O que NÃO está protegido:**
- Se uma skill for habilitada, o código roda no contexto do gateway sem isolamento
- O gateway tem acesso à rede do host (portas mapeadas via bridge Docker)
- Sem `docker.sock` mapeado, o sandbox Docker não consegue criar contêineres — mas isso também significa que a flag `mode: "all"` ficaria sem efeito até o sock ser configurado

**Conclusão:** o sistema está funcional para testes de conversa básica, mas não está pronto para sessões de trabalho real com ferramentas habilitadas.

## Aprendizados de cyber/segurança desta sessão

1. **Fail-closed é bom design.** O OpenClaw recusar subir sem `gateway.mode` explícito é irritante na primeira vez, mas evita que uma config corrompida/infiltrada suba silenciosamente.

2. **Deletar arquivos de repos de terceiros sem entender o build é perigoso.** A "limpeza cirúrgica" deletou templates que o build usa internamente. Sempre auditar o Dockerfile antes de deletar qualquer coisa.

3. **`git status` não mostra o perigo completo.** Arquivos deletados mostram ` D` mas o git não impede que o build quebre — ele só sabe que algo mudou, não o impacto disso.

4. **Documentar o estado inseguro é tão importante quanto documentar o estado ideal.** Uma documentação que só mostra a config segura desejada, mas não registra que o sistema atual está sem sandbox, é uma documentação mentirosa.

5. **O principal vetor de risco agora não é o agente, é o operador.** Com sandbox desligado e sem skills, o agente é inofensivo. O risco é o operador (nós) habilitar uma skill antes de religar o sandbox por descuido.
